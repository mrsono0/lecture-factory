

# IT 기술 강의 교안 글쓰기 프롬프트 향상을 위한 통합 에이전트팀 구성

---

## 1. 설계 철학

두 구성안의 강점을 결합하여 다음 원칙으로 설계합니다:

| 골격 요소 (Claude 기반) | 확장 요소 (GPT 기반) |
|---|---|
| **3계층 파이프라인** (분석→생성→검증) | **학습자 전담 에이전트** (Learner Advocate) |
| **RACI 역할 매트릭스** | **Interface Contract 6종** (공통 규격) |
| **피드백 루프** (검증→총괄→재생성) | **컴플라이언스 에이전트** (Safety & Policy) |
| **오케스트레이터 총괄 구조** | **Core + On-demand 운영 모드** |

---

## 2. 통합 에이전트팀 전체 구조

> **A0(오케스트레이터)** → 분석 계층(A1·A2·A3) → 생성 계층(A4·A5·A6·A7) → 검증 계층(A8·A9) → (선택) 컴플라이언스(A10) → 피드백 루프(A8/A9→A0→재작업)
> 운영 모드: 최소 6인(A0+A1+A2+A4+A5+A8) / 표준 10인(A0~A9) / 고품질 11인(+A10)

## 3. 에이전트별 상세 정의

### A0. 🎯 오케스트레이터 (Orchestrator / 편집장)

| 항목 | 내용 |
|---|---|
| **역할** | 전체 교안 생성 프로세스의 **총괄 조율자이자 최종 의사결정권자** |
| **핵심 책임** | ① 입력된 강의 구성안을 수신하여 각 에이전트에게 **Interface Contract 기반 작업 지시서**를 분배 ② 중간 산출물의 **품질 게이트(Quality Gate)** 통과 여부 판정 ③ 피드백 루프에서 QA 리포트를 수신하여 **재작업 대상·범위를 결정** ④ 에이전트 간 **인터페이스 규격 준수 여부를 감독** |
| **입력** | 강의 구성안 원본, 사용자 추가 요구사항, Interface Contract 6종 |
| **출력** | 작업 지시서(에이전트별), 중간 피드백, 재작업 명령, **최종 프롬프트 번들 승인 판정** |
| **전문성** | 프로젝트 관리, IT 교육 도메인 전반 이해, 품질 관리, 에이전트 오케스트레이션 |
| **판단 기준** | ① 모든 LO가 교안에 반영되었는가 ② 기술적 오류가 없는가 ③ 대상 학습자 수준에 적합한가 ④ 세션 간 논리적 흐름이 자연스러운가 ⑤ Interface Contract 6종이 모두 준수되었는가 |
| **특수 능력** | **프롬프트 번들 목차 자동 생성** — 시스템/유저/컨텍스트/평가기준 프롬프트를 패키지로 구성하여 최종 산출물의 **재사용 가능한 구조**를 보장 |

### A1. 📋 구성안 분석가 (Curriculum Analyst / Outline Parser)

| 항목 | 내용 |
|---|---|
| **역할** | 강의 구성안을 **구조적으로 파싱**하여 모든 에이전트가 즉시 활용 가능한 **정규화된 스펙**으로 변환 |
| **핵심 책임** | 과정 메타데이터(대상, 시간, 환경), 학습목표(LO) 체계, 세션 구조, 시간 배분, 설계 원칙, 차별화 포인트 등을 **Interface Contract의 세션 스펙 카드 표준**에 맞춰 변환 |
| **입력** | 강의 구성안 원본 (마크다운, PDF, 문서 등) |
| **출력** | ① **세션 스펙 카드 세트** (IC-01 형식 준수) ② **LO-세션 매핑 테이블** (커버리지/중복/누락 탐지) ③ **제약조건 목록** ④ **기술 스택·버전 명세** ⑤ **핵심 키워드 리스트** |
| **전문성** | 문서 분석, 정보 추출, IT 교육과정 구조 이해 |
| **특수 능력** | ① **암묵적 전제조건 탐지** — "선수과정 이수자" → 사전지식 수준 추론 ② **시간 배분 현실성 사전 검증** ③ **기술 스택 버전 의존성 자동 파악** ④ **모호성 플래그** — 구성안의 불명확한 부분을 명시적으로 표시하여 A0에게 확인 요청 |
| **IC 준수** | IC-01(세션 스펙 카드 표준), IC-06(용어/표기 규칙) |

### A2. 🎓 교수설계 전문가 (Instructional Design Specialist)

| 항목 | 내용 |
|---|---|
| **역할** | 교육학적 관점에서 **학습 경험 설계 전략**을 수립하고, 세션을 "설명→활동→피드백" 구조로 안정화 |
| **핵심 책임** | 블룸 택소노미 기반 LO 검증, 학습 사이클 설계, 스캐폴딩 전략 수립, 형성평가 프레임워크 설계, 동기부여 요소 배치, **학습활동 패턴 라이브러리** 관리 |
| **입력** | A1의 세션 스펙 카드 세트, A3의 학습자 페르소나 보고서 |
| **출력** | ① **교수설계 전략서** ② **세션별 교안 설계 템플릿** (도입/개념/데모/실습/정리/점검) ③ **학습활동 패턴 라이브러리** (짧은 체크퀴즈, 페어리뷰, 리팩토링 토론 등) ④ **비유/앵커 스토리 방향서** |
| **전문성** | 교수설계(ADDIE, SAM), 블룸 택소노미, 성인학습이론(앤드라고지), 인지부하이론, 건설주의 학습이론 |
| **특수 능력** | ① **기술 난이도 곡선 조절** — IT 개념의 인지부하를 단계적으로 관리 ② **패러다임 전환 브릿지 설계** — 절차적→구조적→OOP 같은 사고 전환 지점에 적합한 교수 전략 ③ **"왜(Why)" 프레이밍** — 모든 세션의 도입부에 학습 동기 부여 구조 보장 |
| **IC 준수** | IC-01(세션 스펙 카드), IC-05(평가 규칙) |

### A3. 👤 학습자 대변인 (Learner Advocate) — ★ GPT 확장

| 항목 | 내용 |
|---|---|
| **역할** | 교안 전체가 **대상 학습자의 수준·심리·동기에 최적화**되도록 학습자 관점의 가드레일을 제공하는 **전담 에이전트** |
| **핵심 책임** | ① 학습자 페르소나 정의 및 관리 ② 난이도 규격 수립(용어 수준, 예제 복잡도, 단계별 설명 규칙) ③ **오개념/불안 포인트 리스트** 작성 ④ 과도한 전제지식 요구 탐지 ⑤ 학습자 동기·심리 모델 기반 콘텐츠 가이드라인 |
| **입력** | A1의 세션 스펙 카드(대상 학습자 정보), 강의 구성안의 대상 정의 |
| **출력** | ① **학습자 페르소나 카드** (배경/수준/동기/불안요소/선수지식 수준) ② **난이도 규격서** (용어 수준 등급표, 예제 복잡도 기준, 설명 단계 규칙) ③ **오개념·불안 포인트 리스트** (예: 변수/타입 혼동, 반복문 두려움, OOP 진입장벽 등) ④ **학습자 여정 감정 맵** (세션별 예상 인지부하·감정 곡선) |
| **전문성** | 학습자 분석, UX 리서치 방법론, 인지심리학, 동기이론(자기결정이론, 기대-가치이론) |
| **특수 능력** | ① **전제지식 갭 탐지** — 교안이 설명 없이 사용하는 개념이 학습자에게 없는 경우를 자동 플래그 ② **감정 곡선 예측** — "이 세션에서 학습자가 좌절할 가능성이 높으므로 격려 요소 배치" 같은 감정 기반 권고 ③ **비전공자 필터** — 모든 교안 텍스트를 비전공자 관점에서 읽어 "이해 불가" 지점을 표시 |
| **IC 준수** | IC-02(프롬프트 파라미터 스키마 — 대상/난이도 필드) |
| **설계 배경** | Claude 원안에서는 이 역할이 A2(교수설계)와 A3(콘텐츠 작성자)에 분산되어 있었으나, **학습자 관점의 깊이 있는 분석이 교안 품질에 결정적 영향**을 미치므로 독립 에이전트로 분리 |

### A4. ✍️ 교안 콘텐츠 작성자 (Content Writer)

| 항목 | 내용 |
|---|---|
| **역할** | 교안의 **본문 텍스트를 실제로 집필**하는 핵심 생성 에이전트 |
| **핵심 책임** | 세션별 교안 본문 작성: 도입부(Hook), 개념 설명, 비유/스토리텔링, 요약, 전환 문장, 강사 스크립트 노트, 학습자 활동 지시문, **"왜 이걸 배우나" 맥락 문장** |
| **입력** | A1의 세션 스펙 카드, A2의 교수설계 전략서, **A3의 학습자 페르소나·난이도 규격·오개념 리스트** |
| **출력** | ① **강사용 가이드** (IC-03 형식) ② **학습자용 핸드아웃** (IC-03 형식) ③ **핵심 개념 설명 블록** ④ **비유/예시 블록** ⑤ **전환 내러티브** ⑥ **세션 도입 스크립트** |
| **전문성** | 기술 문서 작성, 교육용 글쓰기, 스토리텔링, UX 라이팅 |
| **특수 능력** | ① **대상 맞춤 톤 조절** — A3의 페르소나 카드를 기반으로 문체·어휘 수준을 정밀 조절 ② **비유 생성 엔진** — 추상적 IT 개념을 일상적 비유로 변환 (예: "딕셔너리 = 이름표가 붙은 서랍장") ③ **반복 학습 사이클 내러티브화** — 요구사항→프롬프트→생성→리뷰→수정 패턴을 매끄러운 이야기 흐름으로 연결 ④ **A3의 오개념 리스트 반영** — 흔한 착각을 사전에 해소하는 설명 삽입 |
| **IC 준수** | IC-03(출력 포맷 규칙), IC-06(용어/표기 규칙) |

### A5. 💻 기술 코드 전문가 (Technical Code Specialist / Tech SME)

| 항목 | 내용 |
|---|---|
| **역할** | 교안에 포함되는 모든 **코드 예제, AI 프롬프트 템플릿**을 설계·생성하고, **기술적 정확성의 최종 권위자** |
| **핵심 책임** | ① 데모 코드 작성 ② 실습 스타터 코드 + 정답 코드 생성 ③ AI 프롬프트 예시(좋은 예/나쁜 예) 작성 ④ 코드 리뷰 체크리스트 ⑤ 디버깅 시나리오 설계 ⑥ **정확성 체크리스트** (개념 정의, 엣지케이스, 권장/비권장 패턴) ⑦ **실행 환경 설정 가이드** |
| **입력** | A1의 세션 스펙 카드(기술 스택, 환경), A2의 교수설계 전략서(난이도 곡선), **A3의 난이도 규격서** |
| **출력** | ① **코드 예제 세트** (IC-04 형식) ② **AI 프롬프트 템플릿** ③ **코드 리뷰 체크리스트** ④ **디버깅 시나리오** ⑤ **정확성 체크리스트** ⑥ **표준 예제/반례 세트** |
| **전문성** | 다중 프로그래밍 언어, AI 코드 생성 도구 활용, 소프트웨어 설계 패턴, 교육용 코드 설계 |
| **특수 능력** | ① **점진적 복잡도 코드 설계** — 동일 기능을 점차 발전시키는 코드 시리즈 (v1→v2→v3) ② **의도적 오류 코드 생성** — 학습자가 디버깅/리뷰할 수 있는 "교육적으로 틀린 코드" ③ **프롬프트-코드 쌍 생성** — 프롬프트 품질에 따라 코드 품질이 달라지는 비교 예시 세트 ④ **실행 보장** — 모든 코드가 명시된 환경에서 **실제로 실행 가능**하도록 검증 |
| **IC 준수** | IC-04(코드 규칙 — 실행 가능, 버전 명시, 주석 수준) |

### A6. 🎨 시각자료 설계자 (Visual Asset Designer / Concept Visualizer)

| 항목 | 내용 |
|---|---|
| **역할** | 교안의 **시각적 요소**를 설계하여 텍스트 과밀을 방지하고 **한눈에 구조가 보이는** 학습 자료 생성 |
| **핵심 책임** | 개념 관계도, 코드 실행 흐름도, 비교 테이블, 학습 경로 맵, 아키텍처 다이어그램의 구조·내용 정의 |
| **입력** | A4의 교안 본문 초안, A5의 코드 예제 세트 |
| **출력** | ① **시각자료 명세서** ② 텍스트 기반 다이어그램 (Mermaid, ASCII) ③ SVG 마크업 ④ **비교 테이블 구조** ⑤ **도식 요구사항 템플릿** (표/순서도/비교표/개념지도) |
| **전문성** | 정보 시각화, UX 디자인 원칙, 기술 다이어그래밍, 인지심리학적 시각 설계 |
| **특수 능력** | ① **추상→시각 변환** — "OOP의 상속 관계" 같은 추상 개념을 직관적 도식으로 변환 ② **비교 시각화** — 패러다임 비교(절차적 vs 구조적 vs OOP)를 한눈에 파악 가능한 다중 구조 설계 ③ **학습 흐름 시각화** — 커리큘럼 전체 또는 세션 내 학습 사이클을 시각적 로드맵으로 표현 |
| **IC 준수** | IC-03(출력 포맷 규칙) |

### A7. 📝 실습/평가 설계자 (Assessment & Lab Designer) — ★ 기능 분리

| 항목 | 내용 |
|---|---|
| **역할** | 교안의 **실습·과제·퀴즈·루브릭**을 전문적으로 설계하여 **LO 달성을 측정 가능하게** 만드는 에이전트 |

| **IC 준수** | IC-05(평가 규칙), IC-04(코드 규칙) |
| **설계 배경** | Claude 원안에서는 A4(코드 전문가)에 통합되어 있었으나, **실습/평가 설계는 코드 작성과 다른 교육학적 전문성**을 요구하므로 독립 분리 |

### IC-04. 코드 규칙

| 규칙 | 기준 |
|---|---|
| **실행 가능성** | 모든 코드는 `tech_stack`에 명시된 환경에서 **복사-붙여넣기로 즉시 실행** 가능해야 함 |
| **버전 명시** | 파일 상단 또는 주석에 `# Python 3.11+ | 패키지: requests==2.31.0` 형태로 명시 |
| **주석 수준** | 데모 코드: **모든 핵심 라인에 한글 주석** / 실습 스타터: **TODO 주석으로 작업 위치 표시** |
| **함수 분해** | 10줄 이상의 로직은 함수로 분리, 함수명은 동사+명사 (예: `calculate_score`) |
| **입출력 방식** | `input()` 사용 최소화, 가능하면 **하드코딩된 테스트 데이터**로 데모 |
| **예외처리** | 교육 목적에 필요한 최소한의 try-except만 포함 (과도한 방어 코드 지양) |
| **라이선스** | 외부 라이브러리 사용 시 라이선스 호환성 명시 (A10 연계) |

### IC-05. 평가 규칙

| 규칙 | 기준 |
|---|---|
| **LO 직접 매핑** | 모든 평가 문항/과제는 **최소 1개 LO와 명시적으로 연결** |
| **난이도 단계화** | 기본(60%) + 도전(30%) + 심화(10%) 비율 권장 |
| **루브릭 필수** | 채점 가능한 과제에는 **3단계 이상 루브릭** 첨부 (미흡/충족/우수) |
| **과정 평가 포함** | 정답 여부뿐 아니라 **문제 해결 과정**(코드 구조, 변수명, 주석)도 평가 |
| **자기평가 도구** | 각 세션 종료 시 학습자 자기점검 체크리스트 1개 이상 |
| **형성평가 빈도** | 45분당 최소 1회의 형성평가(퀴즈, 실습, 토론) |

### IC-06. 용어/표기 규칙

| 규칙 | 기준 | 예시 |
|---|---|---|
| **용어집 우선** | 교안 전체에서 동일 개념은 **용어집에 정의된 하나의 명칭**만 사용 | "딕셔너리" (O) vs "사전"/"딕트" (X) |
| **영문 병기** | 한글 용어 첫 등장 시 **영문 병기**, 이후는 한글만 | "변수(Variable)는..." → 이후 "변수는..." |
| **약어 규칙** | 약어 첫 등장 시 **풀네임 + 괄호 약어**, 이후 약어만 | "객체지향 프로그래밍(OOP)은..." → 이후 "OOP는..." |
| **코드 내 명칭** | 코드 내 식별자(변수명, 함수명)는 **영문 snake_case** | `student_name` (O) vs `studentName` (△, 별도 명시 시 허용) |
| **외래어 표기** | 국립국어원 표기 기준 우선, IT 관용 표현은 **용어집에 예외 등록** | "파이썬" (O) vs "파이선" (X) |

| **학습자 페르소나 정의** | A | C | C | **R** | I | — | — | — | — | — | — |
| **학습목표 검증** | A | C | **R** | C | — | — | — | — | C | — | — |
| **교수설계 전략 수립** | A | C | **R** | C | I | I | — | — | — | — | — |
| **교안 본문 집필** | A | C | C | C | **R** | C | — | — | — | — | — |
| **코드/프롬프트 생성** | A | C | C | C | — | **R** | — | — | — | — | — |
| **시각자료 설계** | A | — | C | — | C | C | **R** | — | — | — | — |
| **실습/평가 설계** | A | C | C | C | — | C | — | **R** | — | — | — |
| **품질 검증** | A | — | C | C | I | I | I | I | **R** | — | C |
| **프롬프트 최적화** | A | — | — | — | — | — | — | — | C | **R** | — |
| **컴플라이언스 검토** | A | — | — | — | I | I | — | I | C | — | **R** |
| **최종 승인(Go/No-Go)** | **A/R** | — | C | C | — | — | — | — | C | C | C |

> **R** = Responsible(실행) / **A** = Accountable(최종 책임) / **A/R** = A+R(최종 승인 실행) / **C** = Consulted(자문) / **I** = Informed(통보) / **—** = 무관

## 6. 피드백 루프 상세 설계

> **A8(QA 검증)** → 이슈 분류(P0/P1/P2) → **A0(오케스트레이터)** → 해당 에이전트 재작업 → 재검증 루프
> 병렬: **A9(프롬프트 최적화)** — 근본 원인 역추적 → 프롬프트 패치 배포
> (필요 시) **A10(컴플라이언스)** — `compliance_level=public|commercial` 또는 외부 공개/판매/기업 규정 요구 시 수행 → 이슈는 A0 경유 재작업
> 통과 조건: P0 = 0건, P1 잔여 = 0건, 모든 IC 준수 (+필요 시 A10 조건 충족) → A0 최종 승인

## 8. 작업 폴더 구성 및 파일 관리

강의 교안 생성 프롬프트 향상 작업을 시작할 때, **가장 먼저 전용 작업 폴더를 생성**하고 모든 산출물을 해당 폴더 안에서 관리합니다.

### 8-1. 작업 폴더 생성 규칙

| 항목 | 규칙 |
|---|---|
| **폴더명 형식** | `YYYY-MM-DD_강의제목` (시스템 날짜 + 언더스코어 + 강의 제목) |
| **날짜 기준** | 작업 시작 시점의 **시스템 날짜** (예: `2026-02-14`) |
| **강의 제목** | 강의 구성안에 명시된 공식 강의 제목 (공백은 언더스코어로 치환) |
| **생성 위치** | (권장) `강의교안/` 폴더 하위에 생성. `강의교안/`가 없으면 **먼저 생성**한 뒤 그 하위에 생성 |

> **예시**: 2026년 2월 14일에 "AI 네이티브 파이썬 기초" 강의 교안을 작성하는 경우
> → `강의교안/2026-02-14_AI_네이티브_파이썬_기초/`

### 8-2. 작업 폴더 내부 구조

작업 폴더 생성 후, 아래와 같은 하위 구조를 만들어 산출물을 체계적으로 관리합니다.

```
YYYY-MM-DD_강의제목/
├── 00_분석/                  # A1·A2·A3 분석 계층 산출물
│   ├── 세션_스펙_카드.md
│   ├── LO_매핑_테이블.md
│   ├── 학습자_페르소나.md
│   └── 교수설계_전략서.md
├── 01_교안/                  # A4 교안 콘텐츠 본문
│   ├── 강사용_가이드/
│   └── 학습자용_핸드아웃/
├── 02_코드_프롬프트/          # A5 코드 예제 및 프롬프트 템플릿
│   ├── 데모_코드/
│   ├── 실습_코드/
│   └── 프롬프트_템플릿/
├── 03_시각자료/               # A6 다이어그램·표·순서도
├── 04_실습_평가/              # A7 실습 명세·루브릭·퀴즈
│   ├── 실습_명세/
│   ├── 루브릭/
│   └── 퀴즈/
├── 05_검증/                  # A8·A9 QA 리포트 및 프롬프트 최적화 기록
│   ├── QA_리포트/
│   └── 프롬프트_변경_로그/
├── 06_컴플라이언스/           # A10 컴플라이언스 체크 (On-demand)
├── 07_프롬프트_번들/           # A0·A9 최종 프롬프트 번들(재사용 패키지)
│   ├── 시스템_프롬프트/
│   ├── 유저_프롬프트/
│   ├── 컨텍스트/
│   ├── 평가기준/
│   ├── 예시/
│   ├── ALL_IN_ONE.md          # (권장) 단일 파일 완성본(배포/공유/복사 1개용)
│   ├── CHANGELOG.md
│   └── README.md
├── 참고자료/                  # 원본 강의 구성안, 레퍼런스 문서 등
└── README.md                 # 작업 개요, 진행 상태, 산출물 목록
```

### 8-3. 작업 절차

| 단계 | 수행 내용 |
|---|---|
| **① 폴더 생성** | 작업 시작 시 `YYYY-MM-DD_강의제목` 폴더 및 하위 디렉토리를 생성 |
| **② 원본 배치** | 강의 구성안 원본을 `참고자료/` 폴더에 복사 |
| **③ README 작성** | 강의 제목, 대상 학습자, 기술 스택, 작업 시작일, 에이전트 운영 모드 등을 `README.md`에 기록 |
| **④ 단계별 산출물 저장** | 각 에이전트의 산출물을 해당 폴더(`00_분석/` ~ `07_프롬프트_번들/`)에 저장 |
| **⑤ 버전 관리** | 피드백 루프에 의한 수정본은 파일명에 버전 표기 (예: `QA_리포트_v1.md` → `QA_리포트_v2.md`) |
| **⑥ 프롬프트 번들 패키징** | A0가 `07_프롬프트_번들/`에 시스템/유저/컨텍스트/평가기준을 정리하고, A9가 IC-02 파라미터 스키마 정합성과 재사용성을 점검 + **단일 파일 완성본(`ALL_IN_ONE.md`) 생성** |
| **⑦ 최종 산출물 정리** | 품질 게이트 통과 후, 최종 승인된 산출물만 별도 표시하거나 `README.md`에 최종 버전을 명기 |

> [!IMPORTANT]
> 모든 에이전트는 산출물을 **반드시 작업 폴더 내 지정된 하위 디렉토리**에 저장해야 합니다. 작업 폴더 외부에 파일을 생성하지 마십시오.

### 8-4. 실행 런북(프롬프트 향상)

아래 런북은 "교안 자체 작성"이 아니라 **교안 작성용 프롬프트(번들) 향상**에 최적화된 절차입니다.

1) **입력 확정(A0)**: 강의 구성안 원본 + IC-02 파라미터(특히 `target_audience`, `difficulty`, `tech_domain`, `compliance_level`)
2) **정규화(A1)**: IC-01 세션 스펙 카드/LO 매핑/제약·스택/모호성 플래그 생성 → A0가 모호성 해소(질문/가정 확정)
3) **학습자 가드레일(A3)**: 페르소나/난이도 규격/오개념·불안 포인트 확정 → A0가 범위·톤 승인
4) **교수설계(A2)**: 세션 템플릿/활동 패턴/평가 전략 확정 → A0 승인
5) **생성(A4·A5·A6·A7)**: IC-03/04/05 준수 초안 생성(본문/코드/시각/평가)
6) **검증(A8)**: P0/P1/P2 분류 QA 리포트 작성 → P0 또는 P1 잔여가 있으면 A0가 재작업 지시 후 루프
7) **프롬프트 개선(A9)**: QA 이슈를 "프롬프트 결함"으로 역추적하여 `07_프롬프트_번들/` 갱신 + `CHANGELOG.md` 기록
8) **컴플라이언스(선택, A10)**: `compliance_level=public|commercial`이면 수행 → 이슈는 A0 경유 수정
9) **최종 승인(A0)**: 품질 게이트 통과(P0=0, P1=0, IC 준수, 필요 시 A10 통과) 시 Go/No-Go 결정

### 8-5. 단일 파일 완성본(ALL-IN-ONE) 생성 규칙

> 목적: “유저 프롬프트 10개 + 시스템/컨텍스트/평가기준”을 **파일 1개**로 묶어, 전달·재사용·복사-붙여넣기 사용성을 극대화한다.

#### 생성 위치/파일명(고정)

- `07_프롬프트_번들/ALL_IN_ONE.md`

#### 포함 구성(순서 고정)

1) **메타데이터**: 강의 제목, 작성일, 버전, 입력 구성안 파일명, IC-02 기본값  
2) **사용법**: “전체를 한 번에 생성하지 말고, Day/오전/오후 블록을 선택 실행” 안내(토큰/출력 한계 고려)  
3) **SYSTEM 프롬프트**: A4 교안 작성자 시스템 프롬프트(재현 가능한 출력 규칙 포함)  
4) **CONTEXT**: 과정 컨텍스트(대상/스택/LO/난점/가드레일/용어 통일)  
5) **QA 체크리스트**: A8 관점의 최소 점검표(LO 커버리지/시간/난이도/용어/형성평가)  
6) **유저 프롬프트 10개**: Day1~Day5 각각 오전/오후 블록(각 블록은 독립 실행 가능해야 함)

#### 유저 프롬프트 블록 규격(각 블록 공통)

- 블록 제목: `## Day{N} {오전|오후} — {세션 요약}`  
- 반드시 포함:
  - IC-02 파라미터 YAML(기본값 포함)
  - 세션 스펙 요약(시간/LO/활동/산출물)
  - 출력 산출물 2개(강사용 가이드 + 학습자용 핸드아웃) 명시
  - 형성평가 빈도(45분당 1회) 강제
  - 자기검토(최소 3문항)
- 복사-붙여넣기 안정성을 위해 **유저 프롬프트 본문은 코드블록(````text`)으로 감싼다**.

#### 동기화 규칙(중요)

- `유저_프롬프트/`의 모듈(10개) 또는 시스템/컨텍스트/평가기준이 변경되면,
  - 반드시 `ALL_IN_ONE.md`에도 동일 변경을 반영하고,
  - `07_프롬프트_번들/CHANGELOG.md`에 변경 로그를 남긴다.

## 10. 범용성 설계 원칙

| 원칙 | 보장 메커니즘 | 적용 예시 |
|---|---|---|
| **기술 도메인 독립성** | A1이 기술 스택을 자동 추출 → IC-02 파라미터로 전달 → A5·A9가 도메인별 적응 | 파이썬 → JavaScript, 클라우드, 데이터베이스, DevOps |
| **학습자 수준 적응성** | **A3 학습자 대변인**이 페르소나·난이도 규격을 재수립 → A4·A5·A7이 반영 | 비전공 입문 → 현직 개발자 심화 과정 |
| **교수법 유연성** | A2가 구성안의 설계 원칙을 파악하여 다양한 교수법 대응 | 플립러닝, PBL, 강의식, 혼합형 |
| **시간 구조 적응성** | A1이 시간 배분 파싱 → A8이 현실성 검증 | 5일 집중 → 10주 주 1회 → 온라인 비동기 |
| **프롬프트 자기 진화** | A9가 A8 QA 결과 기반으로 프롬프트 지속 개선 | 첫 과정 경험이 다음 과정에 자동 반영 |
| **법적 안전성** | **A10 컴플라이언스**가 배포 전 리스크 사전 차단 | 내부 교육 → 외부 공개 → 상용 판매 |
| **인터페이스 표준화** | **IC 6종**이 에이전트 교체·추가 시에도 호환성 보장 | 새 에이전트 추가 시 IC만 준수하면 즉시 통합 |

# IT 기술 강의 교안 작성 — 에이전트 팀 & 워크플로우

> **이 파일은 AI 코딩/작문 에이전트가 강의 교안 작업 시 자동으로 참고하는 지침서입니다.**
> 상세 정의는 아래 **[📖 상세 참조](#-상세-참조-에이전트-팀-상세-정의-원본)** 섹션을 참조하세요.

---

## 0. 실행 워크플로우 (권장 체크리스트)

> 목적: 역할 정의(참조용)와 별개로, **실제로 “무엇을 어떤 순서로 만들지”를 한 번에 실행 가능**하게 정리한 체크리스트입니다.  
> 운영 방식: **핵심 라인 1회전(초안) → QA 리포트 기반 2회전(수정) → 최종본**이 가장 안정적입니다.

### 0-0. 최우선 원칙(강제): “교안만으로 따라 하기(Self-contained)”

> 목표: **비전공 초보자**가 강의 교안(특히 “학습자용 핸드아웃”)만 보고도 설치/실행/검증/트러블슈팅까지 완주할 수 있게 만든다.  
> 금지: “링크 들어가서 보세요”, “검색해 보세요”, “강사가 설명해줄게요”처럼 **교안 밖 의존**으로 핵심 절차를 생략하지 않는다.

- **외부 링크는 참고용**으로만 둔다. 진행에 필수인 정보(설치/클릭/명령/권한)는 교안 안에 “최소 재현 절차”를 반드시 포함한다.
- 실습/환경설정 섹션은 아래 **단계 템플릿**을 강제 적용한다(각 단계마다 1~2줄이어도 됨):
  - 목적(왜?) → 수행(클릭/명령) → 예상 결과(정상) → 실패 시(가장 흔한 원인/다음 액션) → 성공 기준(체크) → 산출물(로그/스크린샷/파일)
- 실습이 필요하다면(※ **환경 설정/프로그램 설치 제외**) **예제 세트**를 제공한다: **최소 2개(권장 3개)**이며, 난이도/추상도가 높거나 변형 포인트가 많으면 **4개 이상으로 확대**한다.  
  - 권장 구성: 예제 1) 기본(정상 동작) → 예제 2) 변형(조건/파라미터 변경) → 예제 3) 흔한 실수/엣지 + 복구(트러블슈팅)
- **강사용 가이드**에는 형성평가/퀴즈/미니 과제의 **문제+정답+해설**을 함께 기록한다(필요 시 **루브릭/채점 기준** 포함). **정답/해설은 강사용 가이드 “단일 파일” 안에 기록하며, 별도 파일로 분리하지 않는다.** 단, **학습자용 핸드아웃에는 정답/해설/루브릭을 넣지 않는다.** (운영 원칙: 정답/해설이 포함된 강사용 가이드는 기본적으로 **강사 전용**이며 학습자에게 배포하지 않는다.)
- UI 기반 절차는 **메뉴 경로 + 단축키 + 검색 키워드** 3종을 함께 제공한다(IDE/OS 버전 차이 대비).
- OS/셸 차이가 있으면 분기 표기한다(예: Windows PowerShell vs CMD, macOS/Linux Bash).
- “직접 타이핑 최소화” 원칙은 유지한다: 학습자는 **프롬프트로 생성 → 실행/검증 → 리뷰/수정 요청** 흐름으로 진행한다.

**단계 템플릿(권장)**:

```md
### Step N. (행동 이름)
- 목적:
- 실행(클릭/명령):
- 예상 결과:
- 실패 시:
- 성공 기준:
- 기록/산출물:
```

**형성평가(퀴즈/미니 과제) — 강사용 정답/해설 템플릿(권장)**:

```md
### Quiz/미니 과제 N
- (학습자 공개) 문제(학습자에게 제시할 문장):
- (학습자 공개, 선택) 힌트:
- (학습자 공개) 성공 기준(통과 조건/예상 결과):
- (강사 전용) 정답(핵심 1~3줄):
- (강사 전용) 해설(왜 그런가 + 흔한 오해/오답):
- (강사 전용, 선택) 루브릭/채점 기준(통과 기준/배점/채점 포인트):
```

### 0-1. 시작 입력(Inputs)

- **교안 작성 프롬프트 원문** (사용자가 제공한 그대로)
- **원본 소스** (영상/웹/문서/레포/이전 교안 등)
- (권장) **대상 학습자/시간표/실습 환경** (OS/IDE/버전)
- (권장) **현장 제약 조건**: 인터넷 가능 여부, 설치 권한(관리자), 계정/로그인 필요 여부, 회사 PC 보안 정책 등

### 0-2. 폴더 생성 & 원본 보관 (Gate 0)

1. `YYYY-MM-DD_강의제목/` 생성
2. `참고자료/교안작성프롬프트_원문.md`에 **교안 작성 프롬프트 원문을 그대로 저장** (복붙)
3. `참고자료/`에 원본 소스(또는 링크/캡처/파일)를 보관
4. `README.md`에 작업 범위/세션/버전/참고자료 목록 기록
5. (권장) `참고자료/원본소스_목록.md`에 **SRC ID / 파일경로·URL / 접근일 / 버전**을 표로 정리(대용량 원본은 경로로 링크)
6. (조건부) 원본 소스만으로 “교안 단독 재현”이 불가능하면, **인터넷 검색으로 공식 문서/설치 가이드/버전 정보**를 보강하고 `참고자료/`에 저장(또는 링크/캡처)한 뒤 A2에 등록
7. (조건부) UI 절차(설정/메뉴)가 핵심이면 `참고자료/캡처/`에 **스크린샷/캡처**를 보관(교안 본문에는 “텍스트 절차”도 반드시 포함)

> 참고: 이 섹션의 `참고자료/`, `README.md` 등 경로 표기는 기본적으로 **`YYYY-MM-DD_강의제목/` 작업 폴더 하위**를 의미합니다.

> ⚠️ **프롬프트도 “원본 소스”**로 취급합니다. 없으면 추적성(QA)에서 자주 끊깁니다.

### 0-2-1. 참고자료 보강(웹 리서치) — Gate 0.5(조건부)

> 언제 필요한가: 설치/권한/OS 차이/툴 UI 경로/명령어 플래그 등 **원본에 없는 “재현 정보”**가 있을 때

1. 검색 우선순위: **공식 문서(벤더 docs) → 공식 릴리스 노트 → 권위 있는 커뮤니티/재단 문서 → (최후) 블로그**
2. 기록 필수: 검색어, URL, 접근일, 버전(명시된 경우), 핵심 발췌/요약(저작권 준수)
3. 충돌 처리: 출처가 엇갈리면 “가정/주의”로 표시하고, 가장 보수적/재현 가능한 절차를 선택(근거는 A2에 남김)

### 0-3. 핵심 제작 라인 (Gate 1~5)

- **A1 (팩트 패킷)** → `00_소스분석/`에 팩트 패킷 작성 (파일명 예: `A1_팩트패킷.md` 또는 `팩트패킷.md`)
- **A2 (추적 패킷)** → `00_소스분석/`에 추적 패킷 작성  
  - 출처 표에 프롬프트 파일을 `SRC-PROMPT`로 **반드시 등록** (예: `참고자료/교안작성프롬프트_원문.md`, 접근일 포함)
  - 웹 보강자료는 URL/접근일/검색어까지 기록하고, 가능하면 오프라인 사본 또는 캡처를 보관
- **A3 (골격 패킷)** → `01_구조설계/`에 골격 작성 후 **A0 승인**
- **A4 (본문 초안)** → `02_교안본문/`에 교안 본문 작성 (팩트/출처 태그 반영)  
  - 실습/환경설정은 **0-0 단계 템플릿**(목적/수행/예상 결과/실패 루틴/성공 기준/산출물)을 적용
  - 형성평가/퀴즈/미니 과제는 **강사용 가이드에 문제+정답+해설(필요 시 루브릭)**을 포함하고, 학습자용 핸드아웃에는 **문제+힌트/성공 기준만** 포함(정답/해설/루브릭 제외)
- **A5 (기술 검증)** → `03_기술검증/`에서 코드/명령어/절차 재현성 검증  
  - 가능하면 “교안만 보고” 0→1 드라이런(초기 설치/실행)을 1회 수행해 누락/모호함을 잡는다
- **A6 (시각화)** → `04_시각자료/`에 Mermaid/표 산출물 작성 후 본문에 통합
- **A7 (실습 설계)** → `05_실습설계/`에 실습 패킷 작성 (예상 결과/난점-대응/체크리스트/예제 세트: **기본 2~3개+**, 난이도에 따라 확대)
- **A9 (강사 지원)** → `07_강사지원/`에 진행 가이드 작성
- **A8 (QA)** → `06_품질검증/`에서 5대 원칙 전수 검사 후 QA 리포트 발행  
  - 특히 “학습자용 핸드아웃만으로 실습이 가능한가?”를 별도 항목으로 검사

### 0-4. 되돌림 규칙 (Rework Routing)

- **출처/버전/환경이 애매함** → A2 먼저 수정 → 필요 시 A1 재추출
- **팩트 누락/왜곡** → A1 수정 → A4 본문 반영 → A8 재검증
- **코드/명령어 재현 실패** → A5 리포트 → A4 수정(설명/코드) → A5 재검증
- **초보자 단독 재현 실패(교안만으로 진행 불가)** → A7(실습) → A4(본문) → A5(검증) 순으로 수정 → A8 재검증
- **학습목표/시간배분 불일치** → A3 수정(골격) → A4 재조립

> 원본: 아래 상세 참조 섹션 — IT 기술 강의 교안 에이전트 팀 구성 (개선판)

| Agent | 역할명 | 핵심 책임 | 비유 |
|---|---|---|---|
| **A0** | 프로젝트 오케스트레이터 | 프롬프트 파싱 → 작업 분배 → 팩트/추적 패킷 전달 확인 → 최종 조립 | 오케스트라 지휘자 |
| **A1** 🆕 | 소스 추출·구조화 분석가 | 원본 소스에서 핵심 팩트 추출·구조화 → **팩트 패킷** 생성 | 고고학자 |
| **A2** 🆕 | 추적성·레퍼런스 관리관 | 출처·버전·타임스탬프·환경 메타데이터 관리 → **추적 패킷** 생성 | 도서관 사서 |
| **A3** | 커리큘럼 아키텍트 | 학습목표(Bloom)·세션 구조·시간 배분·난이도 곡선 설계 | 건축가 |
| **A4** | 테크니컬 라이터 | 본문 집필(비유→정의→예시 패턴)·어조 관리·팩트 패킷 통합 | 소설가 겸 번역가 |
| **A5** | 기술 검증 엔지니어 | 코드·명령어·설치 절차·버전 호환성 검증 | 건축 감리사 |
| **A6** | 시각화 디자이너 | Mermaid 다이어그램·비교표·개념 관계도 설계 | 인포그래픽 디자이너 |
| **A7** | 학습자 경험 설계자 | 실습 설계·난점 예측·심리적 장벽 해소·성취감 포인트 배치 | UX 리서처 |
| **A8** | 품질 보증 에디터 | 5대 원칙(완전성·명확성·재현성·추적성·원본유지) 검증·최종 교정 | 출판사 교정 편집자 |
| **A9** | 강사 지원 설계자 | 토론 가이드·전환 멘트·시간 관리·비상 대응·레퍼런스 퀵 가이드 | 연극 연출 보조 |

> **핵심 라인**: A0 → A1/A2 → A3 → A4 → A5 → A6 → A8 (직렬)
> **병렬 라인**: A7, A9는 A4 초안 완성 후 투입

## 4. 5대 원칙 검증 매트릭스

| 원칙 | 검증 질문 | 검증 방법 |
|---|---|---|
| **완전성** | 프롬프트의 모든 세부 내용이 반영되었는가? | 프롬프트 항목 ↔ 교안 섹션 1:1 대조 |
| **명확성** | 모호한 표현 없고 파일명/경로/명령어가 정확한가? | 전문 용어 목록 대조 |
| **재현성** | 학습자가 혼자서 100% 따라 할 수 있는가? | **0-0 단계 템플릿**(예상 결과/실패 루틴/성공 기준/산출물) 전수 적용 + OS/셸 분기 + UI 3종(메뉴/단축키/검색) 제공 여부 |
| **추적성** | 모든 사실적 주장에 출처가 명시되어 있는가? | 추적 패킷 ↔ 교안 출처 태그 대조 |
| **원본 유지** | 내용이 임의로 요약/생략/왜곡되지 않았는가? | 팩트 패킷 원문 인용 ↔ 교안 본문 대조 |

## 5. 핵심 서술 패턴

교안 본문 작성 시 아래 순서를 준수합니다:

1. **실생활 비유** → "프롬프트는 식당 주문서와 같아요"

### Agent 0 — 프로젝트 오케스트레이터 (Project Orchestrator)

| 항목 | 내용 |
|---|---|
| **역할 정의** | 전체 교안 제작 프로세스를 총괄하는 **지휘자**. 입력 프롬프트를 분석하여 각 에이전트에게 작업을 분배하고, 산출물 간 정합성을 확인하며, 최종 교안을 조립합니다. |

| **출력** | 각 에이전트별 Task Brief, 최종 조립된 교안 문서 |

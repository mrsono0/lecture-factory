## 1. 프롬프트 구성 4요소 (Persona, Task, Context, Format)

### 정의
AI가 사용자의 의도를 정확히 파악하고 원하는 방식대로 응답을 생성하도록 유도하는 프롬프트의 4가지 기본 뼈대[1]

### 핵심 개념
- **Persona (컨텍스트/역할)**: AI가 응답을 제공할 때 수행할 역할이나 페르소나를 지정 (예: "숙련된 소프트웨어 엔지니어")[1]
- **Task (지시)**: 요약, 번역, 코드 작성 등 AI가 수행해야 할 명확한 단일 명령. 지시가 너무 많으면 AI가 혼란을 겪을 수 있음[1][2]
- **Context (입력 데이터)**: AI가 정보를 처리하는 데 필요한 배경 정보나 실제 텍스트/코드 데이터. ###나 ''' 같은 구분 기호로 지시와 데이터를 분리하면 효과적[1][3]
- **Format (출력 형식)**: 표, 마크다운, JSON 등 원하는 형태의 결괏값을 명시[1][4]

### 실습 예시
"당신은 파이썬 디버깅 전문가입니다(Persona). 다음 코드에서 발생한 문제를 찾고 해결책을 제안해 주세요(Task). 코드: '''[코드 내용]'''. 문제 원인과 수정된 코드를 마크다운 표 형식으로 정리해 주세요(Format)."[1]

## 2. 코드 생성 프롬프트 5대 필수 항목

### 정의
AI에게 코드 작성을 요청할 때 모호함을 없애고 환각(거짓 정보)을 방지하기 위해 반드시 포함해야 하는 5가지 구체적인 지시 사항[6]

### 핵심 개념
1. **기능**: 이 소프트웨어가 해결하려는 문제와 핵심 기능 명시[6]
2. **입력**: 함수나 프로그램이 받는 데이터의 형식(타입) 명시[6][7]
3. **출력**: 어떤 결과값을 반환하거나 출력해야 하는지 명시[6][7]
4. **제약 사항**: 피해야 할 문법이나 사용해야 할 특정 라이브러리, 조건 명시[6]
5. **예외 처리**: 잘못된 입력이나 오류 발생 시 어떻게 대응할 것인지 명시[6][7]

### 실습 예시
"문자열에서 날짜를 파싱하는 파이썬 함수를 개발하세요(기능). YYYY-MM-DD 형식을 입력받아야 하며(입력), datetime 객체를 반환해야 합니다(출력). 외부 라이브러리는 사용하지 마시고(제약 사항), 잘못된 날짜 형식 입력 시 에러 메시지를 출력하세요(예외 처리)."[6][7]

## 3. 점진적 개선(Iterative Refinement)과 CoT / Few-shot

### 정의
복잡한 문제를 한 번의 지시로 해결하려 하지 않고, 예시를 주거나 단계를 나누어 AI의 추론 능력을 극대화하는 고급 프롬프트 기법[8]

### 핵심 개념
- **점진적 개선 (Iterative Refinement)**: 초기 응답을 받은 후, 원하는 답이 나올 때까지 질문을 수정하고 피드백을 주며 대화를 이어나가는 과정[8]
- **Few-shot 러닝 (퓨샷 러닝)**: 지시와 함께 입력 데이터와 출력 결과에 대한 예시를 1~3개 정도 제공하여 AI가 패턴을 학습하도록 유도하는 기법[9]
- **Chain-of-Thought (CoT, 생각의 사슬)**: 복잡한 문제를 한 번에 묻지 않고, 논리적 순서에 따라 여러 단계로 세분화하여 지시하는 방법[10]

### 실습 예시
- **Few-shot**: "1. 입력:[1...], 출력: [0, 0.33, 0.66, 1] / 2. 입력:[3...], 출력: [0, 0.5, 1] / 이 예시들처럼 정수 리스트를 받아 정규화하는 파이썬 함수를 짜줘."[16]
- **CoT**: "1단계: 피보나치 수열 생성기를 기획해. 2단계: 반복문을 사용해. 3단계: 정수 n을 인수로 받는 함수를 작성해. 4단계: 결과를 리스트로 반환해."[17]

## 4. 요구사항과 5W1H (스타버스팅)

### 정의
소프트웨어를 본격적으로 설계하기 전, '무엇을(What)' 만들 것인지 비즈니스 문제와 사용자 니즈를 체계적인 질문을 통해 구체화하는 기획 기법[18]

### 핵심 개념
스타버스팅(Starbursting) 기법을 활용하여 5W1H를 기반으로 요구사항 분해[18]:
- **Who**: 소프트웨어를 사용할 대상 (페르소나)[19]
- **What**: 해결하려는 핵심 문제와 차별점[19]
- **When**: 적절한 사용 시기와 출시/업데이트 일정[19]
- **Where**: 작동할 플랫폼이나 기기[19]
- **Why**: 사용자가 이 서비스를 선택해야 하는 본질적 이유[19]
- **How**: 작동 방식과 사용자 지원 방법[19]

### 실습 예시
"노인들을 위한 건강 추적 앱을 만들고 싶어. 스타버스팅 기법의 5W1H를 적용해서 이 앱에 어떤 핵심 기능들이 포함되어야 할지 질문과 아이디어를 도출해 줘."[19][20]

## 5. SDD (Specification-Driven Development, 명세 주도 개발)

### 정의
대충 느낌만으로 지시하는 '바이브 코딩'의 한계를 극복하기 위해, AI에게 코드를 시키기 전에 먼저 '무엇을' 만들지 명확한 문서(명세)로 정의하는 개발 방법론[21][22]

### 핵심 개념
- **명세 우선 (Spec-First)**: 코드보다 설계도(명세)를 먼저 작성하여 AI가 임의로 해석할 여지를 줄임[23]
- **역할 분리**: 인간은 '무엇을 만들 것인가(What)'를 정의하는 설계자, AI는 '어떻게 구현할 것인가(How)'를 담당하는 실행자[22][23]
- **점진적 구체화**: 처음부터 완벽한 것이 아니라, 큰 그림부터 시작해 세부 사항을 채워나감[23]

### 실습 예시
"투두리스트 앱 만들어줘"라는 모호한 바이브 코딩 대신, "목표: 할 일 관리 / 기능: 할 일 추가(내용, 마감일 입력 필수), 삭제, 완료 처리 / 데이터: 제목(str), 마감일(date) / 예외: 빈 제목 입력 시 에러"와 같이 구체적인 명세를 먼저 작성한 후 AI에게 구현을 요청[6][24]

## 6. PRD 구조와 기능 명세 (FR)

### 정의
소프트웨어의 전반적인 개요, 타깃 사용자, 주요 기능, 비기능 요구사항 등을 담는 상세한 제품 요구사항 정의서(Product Requirements Document)[25]

### 핵심 개념
- **PRD 구조**: 문서 목적, 제품 개요, 기능적 요구사항(FR), 비기능적 요구사항(NFR), 시스템/데이터 요구사항, 제약 사항 등으로 체계적으로 구성[26]
- **기능 명세 (FR, Functional Requirements)**: 시스템이 수행해야 하는 구체적인 동작. 각 요구사항은 고유 식별자(ID)와 검증 가능한 수용 기준(AC, Acceptance Criteria)을 가져야 추적이 가능[25][27][29]

### 실습 예시
고객 인터뷰(VOC)를 바탕으로 "FR-001 (고객 등록): 시스템은 사용자의 이름, 전화번호, 이메일을 입력받아 저장해야 한다. (제약: 전화번호 형식 검증 필수, 중복 불가)"와 같이 고유 ID와 명확한 수용 기준을 포함한 PRD 스펙 문서를 작성[31][32]

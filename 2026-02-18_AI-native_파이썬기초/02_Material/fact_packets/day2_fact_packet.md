**정의**: 프롬프트의 품질을 결정하는 PTCF 4요소 중, AI에게 역할을 부여하는 Persona(페르소나)와 구체적인 단일 행동을 지시하는 Task(지시).
**핵심 개념**: 
- **Persona**: AI가 응답 시 취해야 할 전문적인 역할 (예: 숙련된 소프트웨어 엔지니어).

**실습 예시**: "당신은 10년 차 파이썬 디버깅 전문가입니다(Persona). 다음 제공되는 코드에서 발생한 논리적 오류를 찾고 해결책을 제안해 주세요(Task)."
**강사 스크립트 포인트**: 페르소나를 부여하면 AI가 내부적으로 참조하는 데이터의 가중치가 달라져 훨씬 더 전문적이고 타겟팅된 답변이 나옴을 강조합니다.

**정의**: 프롬프트 PTCF 4요소 중, 문제 해결에 필요한 배경/입력 데이터를 제공하는 Context와 출력 형태를 결정하는 Format.
**핵심 개념**: 
- **Context**: LLM이 정보를 처리하는 데 필요한 실제 코드나 제약 조건 등의 배경 지식. 구분자(###, ''') 사용 필수.
- **Format**: JSON, 마크다운 표, 불릿 포인트 등 원하는 결괏값의 형태.
**실습 예시**: "코드: '''[코드 내용]''' (Context). 문제 원인과 수정된 코드를 마크다운 표 형식으로 3가지 항목으로 정리해 주세요(Format)."
**강사 스크립트 포인트**: 컴퓨터는 사람처럼 눈치껏 데이터를 구분하지 못하므로, 특수기호(구분자)를 통해 '명령'과 '데이터'를 철저히 분리해야 함을 시각적으로 설명합니다.

### 세션 026: 좋은 프롬프트 vs 나쁜 프롬프트 비교 실험
**📚 참고자료 매핑**: 
- `3 프롤프트 엔지니어링.pdf` (§3.8 좋은 프롬프트 vs 나쁜 프롬프트 비교)

**핵심 개념**: 모호한 프롬프트는 환각(Hallucination)과 파편화된 결과를 유발하지만, 구조화된 프롬프트는 일관되고 재현 가능한 고품질 코드를 생성함.
**실습 예시**: 나쁜 예("고객 데이터 정리하는 코드 짜줘") 입력 시 무작위 결과 확인 -> 좋은 예("데이터 분석가로서[P], 아래 CSV를[C], 나이순으로 정렬해[T]. 주석 포함 블록으로 줘[F]") 입력 후 품질 비교.
**강사 스크립트 포인트**: 학생들에게 두 프롬프트를 Gemini에 직접 복사/붙여넣기 하게 하여 결과의 극명한 차이를 모니터로 직접 확인하게 만드는 '아하 모먼트(Aha-moment)'를 연출합니다.

### 세션 028: 점진적 개선(Iterative Refinement)의 이해
**📚 참고자료 매핑**: 
- `3 프롤프트 엔지니어링.pdf` (§3.10 점진적 개선)
- `gemini-for-google-workspace-prompting-guide-101.pdf` (§반복 개선 전략)

**정의**: 단 한 번의 완벽한 지시를 고민하기보다, 초안을 먼저 생성한 후 대화와 피드백을 통해 결과물을 다듬어가는 프롬프팅 기법.
**핵심 개념**: 3단계 워크플로우: 초안 생성(Draft) -> 결과 평가 및 문제 발견(Evaluate & Feedback) -> 구체적 개선 요청(Refine).
**실습 예시**: 1. "간단한 계산기 코드 작성" -> 2. "나눗셈 예외 처리 추가" -> 3. "UI가 있는 클래스 형태로 리팩토링" 등 대화가 꼬리를 무는 과정 시연.
**강사 스크립트 포인트**: AI와의 작업은 핑퐁 게임(티키타카)입니다. 한 방에 완벽한 코드를 얻으려 하지 말고, 우선 코드를 뽑아낸 뒤 조각하듯 다듬는 것이 AI 시대의 개발 방식임을 설명합니다.

### 세션 030: 고급 프롬프트 기법: CoT와 Few-shot
**📚 참고자료 매핑**: 
- `3 프롤프트 엔지니어링.pdf` (§3.5 CoT, §3.6 Few-shot 프롬프팅)

**정의**: 복잡한 논리 구조를 해결하기 위해 AI의 추론 능력을 극대화하는 두 가지 핵심 고급 기법(생각의 사슬, 소수 예시 학습).
**핵심 개념**: 
- **Chain-of-Thought (CoT)**: 복잡한 문제를 1단계, 2단계 논리적 순서로 나누어 지시하거나 "단계별로 생각해(Step-by-step)"라고 유도함.
- **Few-shot Learning**: 입력 데이터와 원하는 출력의 패턴을 보여주는 1~3개의 고품질 예시를 제공하여 작동 방식을 학습시킴.
**실습 예시**: CoT: "1단계 피보나치 기획, 2단계 반복문 설계..." / Few-shot: "입력:[1,2,3] -> 결과:6 / 입력:[4,5] -> 결과:9. 이 패턴으로 다음 리스트를 처리해."
**강사 스크립트 포인트**: AI가 복잡한 수학 공식이나 비즈니스 로직에서 헛소리(환각)를 할 때, 예시 하나(Few-shot)를 주거나 절차를 쪼개주는(CoT) 것이 특효약임을 알려줍니다.

### 세션 033: SDD(Specification-Driven Development)의 이해

- `AI 시대의 서사 v3 - Claude.md` (§SDD)

**정의**: 느낌대로 지시하는 '바이브 코딩(Vibe Coding)'을 넘어, 작성된 명세서(Specification)를 코드 생성의 유일한 진실의 원천으로 삼는 AI 시대 방법론.
**핵심 개념**: 인간은 명세(What)를 기획하고, AI는 구현(How)을 실행함. 바이브 코딩의 5대 한계(맥락 증발, 파편화, 기술 부채, 아키텍처 표류, 테스트 불가)를 SDD(Spec -> Plan -> Task -> Code)로 극복.
**실습 예시**: "투두리스트 대충 이쁘게 짜줘" 대신, 명확하게 정의된 기능, 데이터, 예외 규칙이 담긴 spec.md 문서를 작성한 뒤 이를 AI에게 통째로 주입.
**강사 스크립트 포인트**: 바이브 코딩은 처음 10분은 재밌지만 1시간 뒤엔 스파게티 코드가 됩니다. 진정한 AI 엔지니어는 완벽한 명세서로 AI를 지휘하는 오케스트라 지휘자임을 역설합니다.

### 세션 036: PRD의 핵심 구조와 기능 명세(FR) 작성법
**📚 참고자료 매핑**: 
- `3 프롤프트 엔지니어링.pdf` (§3.9 코드 생성 프롬프트 특화 기법)
- `7 기획.pdf` (§7.3 PRD 구조)

**정의**: PRD의 꽃인 기능 요구사항(FR, Functional Requirements)과 비기능 요구사항(NFR)을 식별자(ID)를 통해 체계적으로 문서화하는 방법.
**핵심 개념**: 모든 기능은 추적 가능한 ID(예: FR-01)를 가지며, 반드시 달성해야 하는 명확한 '수용 기준(Acceptance Criteria)'을 동반해야 함. NFR은 성능이나 제약조건을 명시.
**실습 예시**: "FR-01 (회원가입): 사용자는 이름과 연락처로 가입한다. (수용기준: 연락처 중복 시 경고 출력) / NFR-01 (제약): 외부 라이브러리 없이 기본 파이썬만 사용."
**강사 스크립트 포인트**: 고유 ID(FR-01)를 부여하면 AI와 대화할 때 "너 FR-01 기능에서 수용 기준 빼먹었잖아"라고 아주 정밀한 피드백이 가능함을 데모로 보여줍니다.
